## 6.5 智能体间互操作协议

当一个系统里存在多个智能体（来自不同团队、不同系统、甚至不同组织）时，仅靠“把提示词写好”无法让它们稳定协作。你需要一类**智能体间互操作协议**，把“发现、委派、回传、审计”做成可移植的接口。

### 6.5.1 为什么需要互操作协议

多智能体协作的常见挑战包括：

- **点对点集成爆炸**：每新增一个智能体，都要为既有智能体增加对接逻辑。
- **供应商/实现锁定**：更换智能体实现会牵连大量调用方。
- **通信不一致**：消息格式、任务状态与错误语义各说各话。
- **安全碎片化**：身份认证、授权与审计分散，治理困难。
- **不可发现**：上游智能体无法“自动找到”合适的下游能力。

互操作协议的目标是提供一个统一层，让智能体能够以一致方式：

1. 发现对方的能力
2. 提交任务并追踪状态
3. 在长任务中持续回传进度
4. 在安全边界内传递上下文并完成审计

### 6.5.2 两类协议：智能体互操作与工具连接

工程上常见两类“连接协议”，解决的问题不同：

| 维度 | 工具连接协议 | 智能体互操作协议 |
|------|--------------|------------------|
| **连接对象** | 智能体 ↔ 工具/数据源 | 智能体 ↔ 智能体 |
| **协议目的** | 让智能体获得外部能力 | 让智能体协作完成任务 |
| **交互粒度** | 原子操作与资源访问 | 任务级工作单元与状态机 |
| **核心抽象** | 资源、工具、提示模板 | 能力描述、任务、进度、审计 |
| **生命周期** | 请求-响应为主 | 可能跨秒/分钟/小时的任务流 |

### 6.5.3 协议能力结构

一个实用的智能体互操作协议通常包含以下组件。

#### 1. 能力描述

每个智能体需要发布一个“能力描述文档”，用于：

- 说明智能体身份与用途
- 列出可接受的任务类型
- 给出输入/输出结构（例如 JSON Schema）
- 声明认证方式与调用端点

示意如下：

```json
{
  "name": "HR Assistant",
  "description": "处理员工入职、离职、请假等事务",
  "capabilities": [
    {
      "name": "onboard_employee",
      "description": "处理新员工入职流程",
      "inputSchema": {"type": "object", "properties": {"employee_name": {"type": "string"}}},
      "outputSchema": {"type": "object", "properties": {"employee_id": {"type": "string"}}}
    }
  ],
  "authentication": {"type": "oauth2"},
  "endpoint": "<endpoint>"
}
```

#### 2. 任务模型与上下文透传

智能体之间的协作应该以“任务”为核心单元。为了避免信息在跨节点传递时丢失结构或丧失追踪能力，工程上强烈建议复用 [第 3.6 章](../03_memory/3.6_context_engineering.md) 介绍的 `ContextPack` 结构体来承载数据包，并确保 `trace_id` 全链路透传：

- `id`：任务唯一标识
- `type`：对应能力类型
- `status`：任务状态机（如 `queued` / `in_progress` / `done` / `failed` / `canceled`）
- `payload`：结构化的 `ContextPack` 数据包（包含 system, tools, memory, evidence 等上下文视图）
- `created_at` / `updated_at`：时间戳
- `trace_id`：**核心字段**，必须与 `ContextPack.trace_id` 绑定，串联起上游的规划节点与下游的执行日志。

示意如下：

```json
{
  "id": "task-12345",
  "type": "onboard_employee",
  "status": "in_progress",
  "created_at": "<timestamp>",
  "trace_id": "req-9876-abc",
  "payload": {
    "system": "处理入职逻辑的系统提示词...",
    "memory": [{"role": "user", "content": "张三入职"}],
    "evidence": ["policy_hr_v2_1"]
  }
}
```

#### 3. 进度回传

长任务需要支持“增量回传”，典型形态包括：

- 轮询：客户端按间隔拉取任务状态
- 流式：服务端推送进度事件（例如 Server-Sent Events 或 WebSocket）

关键是事件语义要稳定：`progress`、`checkpoint`、`error`、`final_output` 等。

#### 4. 安全与审计

互操作协议必须能表达：

- 调用方身份与租户归属
- 授权范围（能做什么、不能做什么）
- 审计事件（谁在何时触发了什么任务，传递了哪些上下文）

建议把“敏感上下文”与“可审计元数据”分离传递，避免把凭证或隐私信息直接塞进任务输入。

### 6.5.4 设计建议

- **先统一数据结构**：能力描述、任务状态、错误语义、审计字段。
- **再统一交互方式**：请求-响应、轮询、流式事件。
- **最后再做生态**：目录、发现服务、跨组织信任体系。

### 6.5.5 与工具连接协议的关系

智能体互操作协议与 [4.3 节](../04_tools/4.3_mcp.md) 介绍的工具连接协议（如 MCP）是 **互补而非竞争** 的关系。

| 维度 | 工具连接协议 | 智能体互操作协议 |
|------|------------|----------------|
| **解决什么** | 智能体如何调用外部工具与数据 | 智能体之间如何发现、委派与协作 |
| **交互对象** | 工具服务（Server） | 其他智能体（Agent） |
| **生命周期** | 单次请求-响应 | 可能跨越分钟/小时的任务流 |
| **核心字段** | Resources, Tools, Prompts | Capabilities, Tasks, Progress, Audit |
| **安全焦点** | 路径白名单、最小权限、沙箱 | 身份认证、授权范围、跨组织信任 |

**协同模式**：在实际系统中，一个“远程智能体”既可以通过互操作协议接收任务，也可以通过工具连接协议访问本地工具和数据源。两类协议在 `trace_id` 与审计字段上应保持对齐，确保跨节点的可观测性。


### 6.5.6 A2A与MCP协议对比与实战

#### 协议概览

随着多智能体系统的复杂性增加，跨框架互操作和智能体间通信的标准化变得至关重要。以下对比表展示了两种核心协议的特性差异：

```
╔════════════════╦════════════════╦════════════════╗
║    特性        ║      A2A       ║      MCP       ║
╠════════════════╬════════════════╬════════════════╣
║ 设计目标       ║ Agent间协作    ║ LLM与工具交互  ║
║ 通信方式       ║ 请求-响应      ║ 双向流        ║
║ 部署方式       ║ 网络分布式     ║ 本地客户端     ║
║ 适合场景       ║ 大型Agent网络  ║ 工具集成       ║
║ 复杂度         ║ 高             ║ 中             ║
║ 学习曲线       ║ 陡峭           ║ 缓和           ║
║ 互操作性       ║ Agent间通用    ║ 工具生态统一   ║
║ 成熟度(2026)   ║ v1.0 正式版    ║ 即将成为RFC    ║
║ 企业采用(2026) ║ 100-200家企业  ║ 500+家企业     ║
║ 采用阶段       ║ 早期采纳       ║ 成长阶段       ║
╚════════════════╩════════════════╩════════════════╝
```

注：A2A协议采用情况：2025年底约50家企业试用，2026年初估计100-200家，处于早期采纳阶段。相比之下MCP由于与Claude等主流LLM的集成，企业采用规模更广（500+），处于快速成长阶段。

#### A2A Protocol（Agent-to-Agent）

**A2A的设计目标**是解决异构Agent系统的互操作问题。以下代码演示了A2A消息格式的标准实现和Agent间的协议通信：

```python
# ═════════════════════════════════════════════════════════════
# A2A消息格式标准
# ═════════════════════════════════════════════════════════════

import json
import uuid
from datetime import datetime

class A2AMessage:
    """A2A协议的消息格式"""

    def __init__(
        self,
        agent_id: str,
        message_type: str,
        payload: dict,
        priority: str = "normal"
    ):
        self.protocol_version = "1.0"
        self.agent_id = agent_id
        self.message_type = message_type
        self.message_id = str(uuid.uuid4())
        self.timestamp = datetime.utcnow().isoformat()
        self.priority = priority
        self.payload = payload

    def to_json(self) -> str:
        """转换为A2A标准JSON格式"""
        return json.dumps({
            "protocol_version": self.protocol_version,
            "agent_id": self.agent_id,
            "message_type": self.message_type,
            "message_id": self.message_id,
            "timestamp": self.timestamp,
            "priority": self.priority,
            "payload": self.payload
        })


# 示例：Agent A向Agent B委派任务
task_request = A2AMessage(
    agent_id="agent_acme_001",
    message_type="task_request",
    payload={
        "task_id": "task_uuid_001",
        "task_description": "分析销售数据并生成预测",
        "required_capabilities": ["data_analysis", "forecasting"],
        "input_data": {
            "dataset_url": "s3://bucket/sales_data.csv",
            "time_period": "2025-01"
        },
        "deadline": "2026-03-10T15:00:00Z",
        "callback_url": "https://requester.example.com/a2a/callback"
    },
    priority="high"
)

print(task_request.to_json())


# ═════════════════════════════════════════════════════════════
# A2A Agent的实现示例
# ═════════════════════════════════════════════════════════════

import aiohttp
from typing import Optional, Dict, Any

class A2AAgent:
    """支持A2A协议的Agent"""

    def __init__(self, agent_id: str, capabilities: list[str]):
        self.agent_id = agent_id
        self.capabilities = capabilities
        self.message_queue = []

    async def handle_incoming_request(self, message: Dict[str, Any]):
        """处理来自其他Agent的请求"""

        message_type = message.get("message_type")

        if message_type == "task_request":
            return await self.handle_task_request(message)
        elif message_type == "capability_query":
            return await self.handle_capability_query(message)
        else:
            return {"status": "error", "message": "Unknown message type"}

    async def handle_task_request(self, message: Dict[str, Any]) -> Dict:
        """处理任务请求"""

        task = message["payload"]

        # 检查是否有能力处理
        has_capability = any(
            cap in self.capabilities
            for cap in task.get("required_capabilities", [])
        )

        if not has_capability:
            return {
                "status": "rejected",
                "reason": "No matching capability"
            }

        # 执行任务
        try:
            result = await self.execute_task(task)
            return {
                "status": "completed",
                "task_id": task["task_id"],
                "result": result
            }
        except Exception as e:
            return {
                "status": "failed",
                "error": str(e)
            }

    async def delegate_to_agent(
        self,
        target_agent_id: str,
        task_description: str,
        required_capabilities: list[str],
        callback_url: str
    ) -> Optional[str]:
        """向其他Agent委派任务"""

        request_message = A2AMessage(
            agent_id=self.agent_id,
            message_type="task_request",
            payload={
                "task_id": str(uuid.uuid4()),
                "task_description": task_description,
                "required_capabilities": required_capabilities,
                "callback_url": callback_url
            }
        )

        # 发送请求
        async with aiohttp.ClientSession() as session:
            async with session.post(
                f"https://{target_agent_id}/a2a/receive",
                json=json.loads(request_message.to_json())
            ) as response:
                if response.status == 202:  # Accepted
                    return await response.json()
                else:
                    return None

    async def execute_task(self, task: Dict) -> Any:
        """执行具体任务"""
        # 实现具体的业务逻辑
        pass
```

#### MCP Protocol（Model Context Protocol）

**MCP的设计目标**是统一工具和资源的访问接口。下面演示了如何在MCP服务器中定义工具，以及在Claude中集成使用：

```python
from mcp.server import Server
from mcp.types import Tool, TextContent, ToolResult

# 创建MCP服务器
server = Server("my_data_tools")

@server.tool()
async def query_database(query: str, database: str = "production") -> str:
    """
    查询数据库

    Args:
        query: SQL查询语句
        database: 数据库名称（production或staging）

    Returns:
        查询结果的JSON字符串
    """
    # 实现数据库查询
    result = await async_execute_query(query, database)
    return json.dumps(result)


@server.tool()
async def generate_report(data: dict, format: str = "markdown") -> str:
    """
    基于数据生成报告

    Args:
        data: 报告数据
        format: 输出格式

    Returns:
        格式化的报告
```

#### 选择指南：何时用A2A，何时用MCP

以下决策树帮助开发者根据场景选择合适的协议：

```
决策树：

┌─ 需要多个独立Agent协作？
│  └─ 是 → A2A ✓
│  └─ 否 → 继续
│
├─ 需要统一的工具生态？
│  └─ 是 → MCP ✓
│  └─ 否 → 继续
│
├─ 系统在单机还是分布式？
│  ├─ 单机（同一服务器）→ MCP ✓
│  └─ 分布式（多服务器/云）→ A2A ✓
│
└─ 优先考虑什么？
   ├─ 简化开发 → MCP（学习曲线缓）
   ├─ 企业级部署 → 两者结合（MCP处理工具，A2A处理Agent协作）
   └─ 开源生态 → MCP（生态更成熟）
```

### 6.5.7 跨框架互操作性最佳实践

#### 跨框架Agent迁移

从一个框架迁移到另一个框架时，关键是抽取核心业务逻辑，并将其适配到新框架的架构模式中。下面展示从AutoGen迁移到LangGraph的实际步骤：

```python
# ════════════════════════════════════════════════════════════
# 第1步：分析AutoGen Agent结构
# ════════════════════════════════════════════════════════════

# 原AutoGen Agent
class OldAutoGenAgent(ConversableAgent):
    def __init__(self, name, system_message, llm_config):
        super().__init__(
            name=name,
            system_message=system_message,
            llm_config=llm_config,
            code_execution_config={"use_docker": True}
        )


# ════════════════════════════════════════════════════════════
# 第2步：转换为LangGraph兼容的结构
# ════════════════════════════════════════════════════════════

from langgraph.graph import StateGraph

# 定义标准化状态
class MigrationState(TypedDict):
    agent_name: str
    messages: list
    output: str

# 创建节点（对应原有的Agent逻辑）
def agent_node(state: MigrationState) -> dict:
    # 复用原Agent的核心逻辑
    response = original_agent.generate_reply(
        messages=state["messages"]
    )
    return {
        "messages": [...state["messages"], response],
        "output": response
    }

# 构建图
workflow = StateGraph(MigrationState)
workflow.add_node("agent", agent_node)
workflow.add_edge(START, "agent")
workflow.add_edge("agent", END)

app = workflow.compile()


# ════════════════════════════════════════════════════════════
# 第3步：统一工具定义
# ════════════════════════════════════════════════════════════

# 原工具定义可能不兼容，统一为MCP标准
def convert_tool_to_mcp_format(old_tool) -> dict:
    """将旧格式工具转换为MCP标准"""
    return {
        "name": old_tool.name,
        "description": old_tool.description,
        "input_schema": {
            "type": "object",
            "properties": old_tool.input_spec,
            "required": old_tool.required_inputs
        }
    }

# 使用统一的工具格式
unified_tools = [convert_tool_to_mcp_format(t) for t in old_tools]
```

#### 协议互操作适配层

当需要让使用不同协议的系统互相通信时，适配器模式可以在两个协议之间进行转换。下面实现了A2A和MCP之间的协议适配：

```python
# ═════════════════════════════════════════════════════════════
# 构建A2A ↔ MCP 适配器
# ═════════════════════════════════════════════════════════════

class ProtocolAdapter:
    """A2A与MCP之间的协议适配器"""

    @staticmethod
    def a2a_to_mcp(a2a_message: dict) -> dict:
        """将A2A消息转换为MCP格式"""
        return {
            "jsonrpc": "2.0",
            "id": a2a_message["message_id"],
            "method": "tools/call",
            "params": {
                "name": a2a_message["payload"]["task_description"],
                "arguments": a2a_message["payload"]["input_data"]
            }
        }

    @staticmethod
    def mcp_to_a2a(mcp_response: dict, requester_id: str) -> dict:
        """将MCP响应转换为A2A格式"""
        return {
            "protocol_version": "1.0",
            "agent_id": "adapter_agent",
            "message_type": "task_result",
            "message_id": str(uuid.uuid4()),
            "payload": {
                "result": mcp_response.get("result"),
                "requester_id": requester_id
            }
        }


# ═════════════════════════════════════════════════════════════
# 使用适配器
# ═════════════════════════════════════════════════════════════

adapter = ProtocolAdapter()

# Agent A（使用A2A）发送任务
a2a_request = {
    "protocol_version": "1.0",
    "agent_id": "agent_a",
    "message_type": "task_request",
    "message_id": "msg_001",
    "payload": {
        "task_description": "query_database",
        "input_data": {"query": "SELECT * FROM users"}
    }
}

# 通过适配器转换为MCP格式
mcp_request = adapter.a2a_to_mcp(a2a_request)

# 调用MCP服务
# mcp_result = mcp_server.call_tool(...)

# 转换回A2A格式返回
a2a_response = adapter.mcp_to_a2a(mcp_result, "agent_a")
```
