## 8.6 平台级产品：平台接口与工具使用

除了开源框架，许多模型平台也提供“托管式智能体”能力：包括持久化会话、内置工具、文件处理、以及受控的运行环境。但不同平台对“状态管理”“工具编排”和“执行边界”的分工并不相同，本节更关注这些稳定的设计模式，而不是某个短周期版本号。

### 8.6.1 托管式智能体接口（形态）

平台级接口通常提供：

- **会话/线程**：持久化对话与状态，减少应用侧的状态管理工作。
- **内置工具**：如代码执行、文件检索、结构化输出等，降低工程门槛。
- **托管运行**：平台负责调度、限流与部分安全策略。

伪代码示例：

```python
client = PlatformClient()
agent = client.create_agent(instructions="...", tools=["code_exec", "file_search"])
thread = client.create_thread()
client.add_message(thread, role="user", content="分析上传的数据")
run = client.run_agent(thread, agent)
result = client.wait(run)
print(result.output_text)
```

#### 案例：Claude Code SDK

Anthropic 的 Claude Code SDK 是托管式智能体的一个代表性实现，它把 Claude Code 作为可编程的子进程/运行时来使用，使其能够在受控环境中执行文件编辑、终端命令、代码执行与 MCP 扩展等工作流。

**核心特性**：
- **计算机访问**：支持 Bash、文件读写、代码执行等能力，让智能体在受控环境中完成复杂任务
- **权限控制**：支持 `allowedTools`、`disallowedTools`、`permissionMode` 等细粒度权限配置
- **可插拔工具与协议**：支持 Model Context Protocol（MCP）标准，便于接入第三方工具和数据源
- **子智能体支持**：可将专门角色拆分为子智能体，降低上下文耦合

**与传统托管式接口的差异**：相比完全由平台封装的接口，Claude Code SDK 更像“托管运行时 + 可控工具链”，开发者既能复用平台能力，也能保留较细的工具与权限控制。

#### 其他主流托管式平台

除 Anthropic 外，OpenAI 与 Google 也提供面向智能体的接口，但它们更适合从“状态管理方式”而不是“单一产品名”来理解：

- **OpenAI Responses API**：OpenAI 当前推荐新项目优先考虑的原语。它支持内置工具、结构化输出、多轮交互，并提供多种状态管理方式：`previous_response_id` 可串联多轮响应，`store: true` 可让平台保留响应用于后续引用；如果你需要更持久的会话对象，还可以把它和 Conversations API 结合起来。换句话说，它不是“必须开发者自己管理全部对话历史”的旧式接口。

- **Google Interactions API**：Google 在 ADK 生态中推出的新一代多轮交互原语，更适合把“raw model inference”和“managed agent”统一在一个接口层里理解。它更像 `generateContent` 的演进和补充，而不是单纯的聊天接口替代品；不过当前官方仍将其标为 beta/preview，并明确提示生产场景优先参考 `generateContent`。因此，做平台选型时应把它视为值得关注的新接口，而不是已经完全稳定的默认主路径。

这些平台与 Claude Code SDK 的核心差异在于：前者更偏“统一交互原语”，后者更偏“托管运行时 + 权限化工具链”。工程选型时，建议先看状态由谁托管，再看工具和权限由谁控制。

### 8.6.2 直接工具调用接口（形态）

另一类平台更强调“模型 + 工具”的直接组合：应用侧自己管理状态，但能更灵活地定义工具与执行策略。

常见模式包括：

- **声明工具 Schema**：用 JSON Schema/类型系统描述工具输入输出。
- **执行与回填**：模型产生工具调用，应用执行后把结果回填到上下文。
- **可插拔执行环境**：需要时引入沙箱（代码执行/浏览器操作），并用权限策略约束风险。

伪代码示例：

```python
client = ModelClient(model="<MODEL>")
tools = [{"name": "get_stock_price", "input_schema": {...}}]
resp = client.generate(messages, tools=tools)
if resp.tool_calls:
    outputs = run_tools(resp.tool_calls)
    resp = client.generate(messages + [resp] + outputs, tools=tools)
```

### 8.6.3 主流平台能力矩阵

以下矩阵聚焦于 **平台接口形态** 与 **状态管理方式** 的差异，补充 [8.1 节](8.1_ecosystem.md) 的框架生态分类。框架间的全面横向对比详见第 8.1 节的选型指南。

| 平台 | 形态 | 状态管理 | 内置工具支持 | 控制粒度 | 学习曲线 | 适用场景 |
|------|------|---------|------------|---------|---------|---------|
| Claude Code SDK | 托管式运行时 | 平台托管 | 强（文件、代码、MCP、权限） | 高 | 中等 | 复杂自动化、代码执行 |
| OpenAI Responses API | 直接工具调用 + 可选状态托管 | 应用自管或平台保留 | 强（内置工具、computer use、MCP） | 高 | 中等 | 灵活编排、agentic apps |
| Google Interactions API / ADK | 直接工具调用 + 多轮交互（仍属 preview/beta） | 应用自管或服务端串联 | 强（原生 agent 入口、内置工具） | 高 | 中等 | 信息检索、多模态任务 |
| Dify | 低代码平台 | 平台托管 | 强（工作流内置） | 中 | 低 | 快速原型、可视化开发 |
| LangFlow | 低代码平台 | 平台托管 | 中（LangChain 组件） | 中 | 低 | 教育、早期实验 |
| Flowise | 低代码平台 | 平台托管 | 中（节点式拖拽） | 中 | 低 | 内部工具、快速验证 |

#### 托管式 vs 直接调用接口的对比

| 特性 | 托管式接口 | 直接工具调用接口 |
|------|-----------|------------------|
| 状态管理 | 平台托管（线程/会话） | 应用自管（可定制） |
| 内置能力 | 常提供代码执行/文件检索等 | 通常更轻量、更可插拔 |
| 控制粒度 | 中（平台封装较多） | 高（应用侧编排） |
| 运维负担 | 低 | 中到高（取决于工具与沙箱） |

#### 选型建议

**选择托管式接口的场景**：

- 需要快速原型与较少基础设施投入
- 希望平台承担会话、文件与部分工具能力
- 团队更关注产品闭环而非底层编排

**选择直接工具调用接口的场景**：

- 需要更低延迟或更精细的控制
- 工具体系复杂，执行策略需要高度可定制
- 需要把权限/审计/沙箱与内部系统深度整合

**选择开源框架的场景**：

- 需要完全控制权或本地/私有化部署
- 需要支持多模型与多运行环境
- 合规要求高，需要可审计与可移植

### 8.6.4 小结

平台级产品降低了智能体开发的门槛：

| 形态 | 核心优势 | 适用团队 |
|------|----------|----------|
| 托管式接口 | 集成度高、门槛低 | 快速验证或平台化落地 |
| 直接工具调用 | 灵活可控、可插拔 | 有工程能力与复杂工具需求 |
| 开源框架 | 可移植、可治理 | 生产级与合规要求较高 |

选择建议：先用门槛更低的形态验证价值，再在需要时迁移到更可控、可治理的架构。
